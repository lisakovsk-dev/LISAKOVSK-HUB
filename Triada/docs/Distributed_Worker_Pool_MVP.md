# Техническое задание: Distributed Worker Pool MVP

## Назначение

Данный модуль является следующим этапом развития AI Office.

Цель модуля — обеспечить параллельное выполнение задач, отказоустойчивость и горизонтальное масштабирование без изменения существующей логики Orchestrator и Universal Worker.

Модуль проектируется поверх уже реализованных:

* Orchestrator
* Task Queue
* Universal Worker
* Skills System

---

# Текущее состояние системы

На момент создания документа уже реализованы:

```text
Orchestrator
    ↓
Task Queue
    ↓
Universal Worker
    ↓
Skills
```

Оркестратор умеет:

* принимать запрос;
* выполнять декомпозицию;
* выбирать навыки;
* выбирать модель;
* выбирать инструменты;
* создавать задачи;
* помещать задачи в очередь;
* получать результаты;
* формировать итоговый отчёт.

Universal Worker умеет:

* получать задачу;
* загружать указанные навыки;
* использовать назначенную модель;
* использовать разрешённые инструменты;
* выполнять задачу;
* возвращать отчёт;
* освобождать ресурсы после выполнения.

---

# Цель следующего этапа

Перейти от одного Worker к пулу Worker.

Целевая схема:

```text
Orchestrator
        ↓
Task Queue
        ↓
Worker Pool Manager
        ↓
 ┌─────────────┐
 │ Worker-1    │
 ├─────────────┤
 │ Worker-2    │
 ├─────────────┤
 │ Worker-3    │
 └─────────────┘
        ↓
      Skills
```

---

# Основные задачи

## Регистрация Worker

Каждый Worker должен иметь:

```yaml
worker_id:
worker_name:
status:
profile:
started_at:
last_heartbeat:
```

Пример:

```yaml
worker_id: worker-01
worker_name: Default Worker
status: idle
profile: universal
```

---

# Worker Pool Manager

Создать отдельный компонент:

```text
worker_pool_manager.py
```

Задачи:

* регистрация Worker;
* хранение статусов;
* поиск свободного Worker;
* распределение задач;
* контроль выполнения;
* возврат задач в очередь;
* мониторинг состояния.

---

# Состояния Worker

Поддерживаемые состояния:

```text
idle
loading
working
waiting
completed
failed
offline
```

---

# Heartbeat

Каждый Worker обязан отправлять heartbeat.

Пример:

```yaml
worker_id: worker-01
status: working
timestamp: 2025-01-01T12:00:00
```

Если heartbeat отсутствует дольше заданного интервала:

```text
Worker
    ↓
offline
```

---

# Автоматический возврат задач

Если Worker становится offline:

```text
Worker
    ↓
offline
    ↓
Task Queue
    ↓
requeue
```

Задача возвращается в очередь.

Потеря задач недопустима.

---

# Масштабирование через конфигурацию

Количество Worker должно задаваться через конфиг.

Пример:

```yaml
workers:

  count: 3

  profile: universal
```

Либо:

```yaml
workers:

  - id: worker-01
    enabled: true

  - id: worker-02
    enabled: true

  - id: worker-03
    enabled: true
```

Без изменения кода.

---

# Профили Worker

На первом этапе все Worker используют один шаблон.

Допускается будущая поддержка профилей:

```yaml
profile:
  universal
```

В дальнейшем:

```yaml
profile:
  research
```

```yaml
profile:
  coding
```

```yaml
profile:
  operations
```

Важно:

Профиль не меняет код Worker.

Профиль влияет только на настройки по умолчанию.

---

# Независимость Worker

Каждый Worker является отдельным экземпляром шаблона Universal Worker.

Все используют:

* одинаковый код;
* одинаковую архитектуру;
* одинаковый протокол взаимодействия.

Отличаются только:

* задачами;
* навыки;
* инструментами;
* моделью;
* контекстом.

---

# Совместимость с Render

Архитектура должна поддерживать:

```text
Render
 ├── Orchestrator Service
 ├── Worker Service 1
 ├── Worker Service 2
 └── Worker Service 3
```

Без изменения бизнес-логики.

---

# Будущая совместимость

Модуль должен проектироваться с возможностью перехода на:

* Redis Queue;
* PostgreSQL Queue;
* Celery;
* RabbitMQ;
* Kubernetes;
* Docker Swarm;
* отдельные VPS Worker.

Без изменения API Orchestrator.

---

# Ограничения MVP

Не реализовывать:

* Kubernetes;
* распределённые очереди;
* автоматическое масштабирование;
* специализированные Worker;
* межсерверную синхронизацию.

Оставить точки расширения через интерфейсы и адаптеры.

---

# Главный принцип

Orchestrator остаётся мозгом системы.

Worker остаются универсальными исполнителями.

Worker Pool отвечает за масштабирование и отказоустойчивость.

Ни один Worker не должен быть уникальным или незаменимым.

Любой Worker может быть остановлен, удалён или заменён без влияния на работу всей системы.
