# Архитектура Оркестратора и Universal Worker Pool

## Цель

Построить масштабируемую систему выполнения задач, где Оркестратор управляет работой универсальных исполнителей (Universal Workers), а компетенции загружаются через систему навыков (Skills).

Система должна быть отказоустойчивой, модульной и масштабируемой.

---

# Основная архитектура

```text
Telegram
    ↓
Aiogram Gateway
    ↓
Orchestrator
    ↓
Planner
    ↓
Task Queue
    ↓
Worker Pool
    ↓
Skills
```

---

# Принцип работы

Оркестратор не выполняет задачи самостоятельно.

Оркестратор:

* принимает запрос;
* выполняет декомпозицию;
* определяет необходимые навыки;
* выбирает модель;
* формирует инструкции;
* создаёт задачи;
* помещает задачи в очередь.

---

# Worker Pool

Вместо большого количества специализированных агентов используется пул универсальных исполнителей.

Пример:

```text
Worker-1
Worker-2
Worker-3
```

Все воркеры используют один шаблон.

Различается только назначенная задача.

---

# Universal Worker

Universal Worker не знает:

* архитектуру проекта;
* бизнес-логику;
* клиентов;
* процессы компании.

Worker получает всё необходимое от Оркестратора.

---

# Формат задачи

Пример:

```yaml
task:
  id: task_001

  priority: normal

  model: deepseek-reasoner

  skills:
    - web_search
    - competitor_analysis
    - report_generation

  tools:
    - github
    - notion

  context:
    project: AI Office

  instructions:
    - Найти конкурентов
    - Провести анализ
    - Подготовить отчёт

  expected_output:
    markdown_report
```

---

# Обязанности Worker

После получения задачи Worker обязан:

1. Загрузить указанные навыки.
2. Проверить доступность инструментов.
3. Выполнить задачу.
4. Сформировать отчёт.
5. Вернуть результат Оркестратору.
6. Освободиться для следующей задачи.

---

# Система навыков

Навыки являются независимыми модулями.

Структура:

```text
skills/
├── web_search/
│   └── SKILL.md
│
├── github_analysis/
│   └── SKILL.md
│
├── report_generation/
│   └── SKILL.md
│
├── monetization_scan/
│   └── SKILL.md
│
└── architecture_review/
    └── SKILL.md
```

---

# Формат навыка

Пример:

```yaml
---
name: web_search

description: Поиск информации в интернете

tools:
  - web_search

recommended_model:
  - deepseek-chat
---
```

Далее размещается инструкция выполнения навыка.

---

# Загрузка навыков

Worker не хранит все навыки в памяти.

Алгоритм:

1. Получить задачу.
2. Получить список навыков.
3. Загрузить только необходимые SKILL.md.
4. Выполнить задачу.
5. Освободить память.

Это позволяет сохранять компактный контекст.

---

# Выбор модели

Модель назначается Оркестратором.

Пример:

```yaml
simple_task:
  model: deepseek-chat

analysis_task:
  model: deepseek-reasoner

coding_task:
  model: claude

critical_task:
  model: gpt-4o
```

Worker не выбирает модель самостоятельно.

---

# Масштабирование

MVP:

```text
1 Orchestrator
3 Universal Worker
```

Рабочая конфигурация:

```text
1 Orchestrator
5-10 Universal Worker
```

Крупная система:

```text
1 Orchestrator
20+ Universal Worker
```

---

# Отказоустойчивость

Если Worker недоступен:

```text
Worker-2
↓
Ошибка
↓
Задача возвращается в очередь
↓
Worker-3 забирает задачу
```

Работа системы не должна останавливаться при отказе отдельного Worker.

---

# Принцип шаблона

Все Workers являются экземплярами одного шаблона.

Создание нового Worker не требует написания нового кода.

Достаточно запустить новый экземпляр сервиса.

---

# Будущее развитие

В дальнейшем допускается:

* параллельное выполнение задач;
* распределение задач между серверами;
* автоматическое масштабирование Worker Pool;
* назначение специализированных моделей;
* распределённые очереди задач;
* Kubernetes;
* автономные команды Worker.

При этом архитектура MVP должна оставаться совместимой с будущим развитием.

---

# Главный принцип

Оркестратор думает.

Workers выполняют.

Skills содержат компетенции.

Ни один Worker не должен быть перегружен знаниями или инструкциями, которые не относятся к его текущей задаче.
