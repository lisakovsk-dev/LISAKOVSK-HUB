# AI_Office_Template_MVP.md

# Техническое задание: AI Office Template (MVP)

## Назначение

AI Office Template — универсальная платформа управления задачами, проектами, знаниями, агентами и автоматизациями.

Система должна использоваться как базовый шаблон для создания любых AI-систем без изменения архитектурного ядра.

Основной принцип:

> Конфигурация вместо хардкода.

---

# Цели проекта

Создать платформу, которая:

* управляется через Telegram;
* масштабируется без переписывания архитектуры;
* поддерживает специалистов (агентов);
* поддерживает внешние интеграции;
* использует GitHub как источник истины;
* использует Notion как операционный центр;
* использует Supabase как память и состояние;
* поддерживает MCP;
* разворачивается на Render;
* позволяет заменять любые компоненты через конфигурацию.

# Роли компонентов

## Telegram

Основной пользовательский интерфейс.

Назначение:

* команды;
* диалоги;
* уведомления;
* отчёты.

---

## Aiogram Gateway

Единая точка входа сообщений.

Отвечает за:

* получение сообщений;
* маршрутизацию;
* передачу запросов Оркестратору.

---

## Orchestrator

Главный координатор системы.

Отвечает за:

* анализ запросов;
* планирование;
* декомпозицию задач;
* распределение задач;
* контроль выполнения;
* сбор результатов;
* формирование отчётов.

---

## Planner

Отвечает за:

* декомпозицию задач;
* построение логических цепочек;
* определение зависимостей.

---

## Skill Audit

Отвечает за:

* определение необходимых навыков;
* проверку наличия навыков;
* поиск недостающих компетенций.

---

## Specialist Manager

Отвечает за:

* выбор специалистов;
* запуск специалистов;
* получение результатов.

---

## Report Builder

Отвечает за:

* промежуточные отчёты;
* итоговые отчёты;
* сводки выполнения.

---

# Обязательный цикл работы Оркестратора

Перед выполнением любой задачи Оркестратор обязан:

1. Понять запрос.
2. Определить цель.
3. Собрать недостающую информацию.
4. Выполнить поиск информации при необходимости.
5. Провести декомпозицию.
6. Определить необходимые навыки.
7. Проверить наличие навыков.
8. Сформировать варианты решения.
9. Подготовить план.
10. Показать план пользователю.
11. Дождаться подтверждения.

Без подтверждения выполнение запрещено.

* удаления файлов;

* изменений безопасности;

Потом выполни.

# Специалисты

На этапе MVP специалисты реализуются через инструкции.

Структура:

```text
agents/
├── orchestrator.md
├── architect.md
├── analyst.md
├── recruiter.md
├── auditor.md
├── researcher.md
└── marketer.md
```

Добавление нового файла должно автоматически регистрировать нового специалиста.

Изменение кода не требуется.

---

# GitHub

GitHub является источником истины.

Содержит:

* документацию;
* архитектуру;
* правила;
* инструкции специалистов;
* ADR;
* исходный код.

Пример конфигурации:

```env
GITHUB_OWNER=
GITHUB_REPOSITORY=
GITHUB_BRANCH=
```

Смена репозитория не должна требовать изменения кода.

---

# Notion

Notion является операционным центром.

Содержит:

* проекты;
* задачи;
* встречи;
* клиентов;
* процессы;
* дорожные карты.

Интеграция должна подключаться через MCP и конфигурацию.

---

# Supabase

Supabase является системой памяти.

Содержит:

* память агентов;
* историю действий;
* логи;
* состояние задач;
* пользовательские данные.

Пример:

```env
SUPABASE_URL=
SUPABASE_KEY=
SUPABASE_SCHEMA=
```

---

# MCP

MCP является единым слоем интеграций.

Поддерживает:

* GitHub;
* Notion;
* Google Docs;
* Telegram;
* Render;
* внешние API.

Подключение новых интеграций должно выполняться через конфигурацию.

# MVP без полноценного Multi-Agent

На первом этапе допускается использование одной LLM.

Сценарий:

```text
Сообщение
    ↓
Оркестратор
    ↓
Определение ролей
    ↓
Последовательный вызов специалистов
    ↓
Сбор результатов
    ↓
Формирование отчёта
```

В будущем должна быть возможность перейти на:

* параллельное выполнение;
* распределённую систему специалистов;
* полноценную мультиагентную архитектуру.

Без изменения внешнего API.

# Архитектурные принципы

## Главный принцип

Ни один компонент системы не должен становиться точкой невозврата.

Любой компонент должен иметь возможность:

* быть заменённым;
* быть отключённым;
* быть масштабированным;
* быть вынесенным в отдельный сервис;
* быть перенесённым на другую платформу.

Без переписывания остальной системы.

---

## Принцип перекрытых кранов

Система не должна содержать архитектурных тупиков.

Любой сервис должен иметь замену через адаптер.

Примеры:

### LLM

* OpenRouter
* OpenAI
* Anthropic
* Ollama
* локальные модели

### Storage

* Supabase
* PostgreSQL
* SQLite
* Redis

### Messenger

* Telegram
* Discord
* Slack
* Web UI

### Документация

* GitHub
* Notion
* Google Docs

---

## Принцип конфигурации

Все настройки должны храниться в:

* YAML;
* ENV;
* настройках системы.

Запрещено хардкодить:

* API ключи;
* URL;
* модели;
* идентификаторы;
* названия провайдеров;
* параметры интеграций.

---

## Принцип расширяемости

Добавление нового компонента не должно требовать изменения существующей архитектуры.

Примеры:

Добавление специалиста:

```text
agents/marketer.md
```

Добавление LLM:

```text
providers/llm/new_provider.py
```

Добавление нового канала:

```text
providers/messenger/new_channel.py
```

После регистрации в конфигурации компонент автоматически становится доступным системе.

---

## Принцип независимости инфраструктуры

Поддерживаемые варианты размещения:

* Render
* VPS
* Docker
* Kubernetes
* локальный запуск

Переезд между платформами не должен требовать изменения бизнес-логики.
